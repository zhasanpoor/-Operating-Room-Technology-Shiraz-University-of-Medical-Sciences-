## امکانات

- **کتابخانهٔ عمل‌های جراحی** در دسته‌بندی‌های تخصصی (جنرال، قلب، مغز و اعصاب، ...).
- برای هر عمل: شرح کامل، لیست وسایل، تا دو ویدیو، و فایل ارائه.
- **پخش ویدیو داخل خود سایت** به‌صورت embed — یوتیوب، آپارات و ویمئو.
- **ویرایشگر متن غنی** (Quill) با نوار ابزار فارسی، درج تصویر داخل متن، جدول و لیست.
- **گردش کار محتوا:** نویسنده پیش‌نویس می‌نویسد → برای بررسی می‌فرستد → مدیر
  تأیید/رد/درخواست اصلاح می‌کند → پس از تأیید پست **قفل** می‌شود.
- **گفتگوی نویسنده و مدیر** روی هر پست + سیستم اعلان داخل سایت.
- **داشبورد گرافیکی** مجزا برای مدیر و نویسنده، با نمودار و انیمیشن.
- **سطح‌بندی و تشویق نویسندگان:** برنزی 🥉 / نقره‌ای 🥈 / طلایی 🥇 / الماسی 💎.
- پروفایل کاربر با عکس، بیو و تغییر رمز.
- جستجوی زندهٔ عمل‌ها، پاپ‌آپ راهنمای کاربر تازه‌وارد، حالت تیره.
- صفحات خطای اختصاصی ۴۰۴ و ۵۰۰ به فارسی.

## سطوح کاربری

| سطح | دسترسی |
|-----|--------|
| **مدیر** (`admin`) | دسترسی کامل: مدیریت کاربران و سطوح، تأیید محتوا، گزارش‌های مدیریتی، تنظیمات. |
| **نویسنده** (`editor`) | ساخت و ویرایش **فقط پست‌های خودش**، ارسال برای تأیید. پس از تأیید، پست قفل و غیرقابل‌ویرایش می‌شود. |
| **کاربر** (`user`) | مشاهدهٔ محتوای تأییدشده، ویرایش پروفایل، درخواست ارتقا به نویسنده. |

نکته: تغییر سطح دسترسی **داده‌های قبلی را پاک نمی‌کند**. اگر نویسنده‌ای به کاربر عادی
تنزل و بعداً دوباره نویسنده شود، همهٔ پست‌ها و آمار قبلی‌اش سر جایش است.

## معماری و پشتهٔ فنی

- **بک‌اند:** Node.js + Express.
- **پایگاه داده:** لایهٔ واحد در `db/index.js` که روی دو موتور کار می‌کند —
  - **PostgreSQL** وقتی `DATABASE_URL` تنظیم باشد (توصیه‌شده برای سرور).
  - **SQLite** از طریق `sql.js` در غیر این صورت (توسعهٔ محلی، بدون نیاز به نصب).
  - کوئری‌ها همیشه با جای‌گذار `?` نوشته می‌شوند؛ لایه برای Postgres به `$n` تبدیل می‌کند.
- **فرانت‌اند:** HTML/CSS/JavaScript خالص (بدون فریم‌ورک و بدون مرحلهٔ build).
- **تاریخ شمسی:** `public/js/jalali.js` — یک ماژول مشترک سرور و مرورگر.
- **مهاجرت اسکیما:** `migrations.js` — افزایشی و idempotent؛ با هر بار اجرا فقط
  مهاجرت‌های جدید اعمال می‌شوند و در جدول `schema_migrations` ثبت می‌گردند.

> **چرا `sql.js` و نه `better-sqlite3`؟** بستهٔ `better-sqlite3` نیاز به کامپایل
> بومی دارد که روی محیط استقرار به مشکل می‌خورد. `sql.js` پیاده‌سازی خالص
> جاوااسکریپت است و همه‌جا کار می‌کند.

## راه‌اندازی محلی

پیش‌نیاز: **Node.js ۱۸ یا بالاتر**.

```bash
git clone <repo-url>
cd NiKA_clean
npm install

# فایل نمونهٔ محیط را کپی کنید (اختیاری برای توسعه)
cp .env.example .env

npm run dev      # با nodemon (بارگذاری خودکار)
# یا
npm start        # اجرای ساده
```

سپس مرورگر را باز کنید:

- سایت اصلی: <http://localhost:3000>
- پنل مدیریت: <http://localhost:3000/admin>

**اولین اجرا:** اگر دیتابیس خالی باشد، به‌طور خودکار seed می‌شود (۱۰ دسته‌بندی،
۱۴۳ عمل، محتوای ۱۸ عمل اول) و یک حساب مدیر ساخته می‌شود. رمز تصادفی مدیر **یک بار**
در خروجی کنسول چاپ می‌شود — دنبال خط `ADMIN_PASSWORD` بگردید. برای رمز ثابت،
متغیر `ADMIN_PASSWORD` را تنظیم کنید.

## متغیرهای محیطی

فایل `.env.example` را ببینید. مهم‌ترین‌ها:

| متغیر | توضیح |
|-------|-------|
| `PORT` | پورت سرور (پیش‌فرض ۳۰۰۰). |
| `NODE_ENV` | `development` یا `production`. |
| `DATABASE_URL` | رشتهٔ اتصال PostgreSQL. **اگر تنظیم نشود، SQLite موقت استفاده می‌شود.** |
| `JWT_SECRET` | کلید امضای توکن — حداقل ۳۲ کاراکتر تصادفی. |
| `ADMIN_USERNAME` / `ADMIN_PASSWORD` | حساب مدیر اولیه. |
| `ALLOWED_ORIGINS` | دامنه‌های مجاز CORS در production، جداشده با کاما. |

> **`JWT_SECRET`:** اگر تنظیم نشود، سرور یک کلید تصادفی قوی می‌سازد و اجرا می‌شود،
> اما آن کلید با هر ری‌استارت عوض می‌شود و کاربران باید دوباره وارد شوند. برای
> پایداری نشست‌ها حتماً روی سرور تنظیمش کنید. کلید ضعیف یا لو رفته پذیرفته نمی‌شود.

## استقرار روی Render

پروژه شامل `render.yaml` (Blueprint) است. اگر سرویس را از روی Blueprint بسازید،
Render خودش دیتابیس Postgres و `JWT_SECRET` و `DATABASE_URL` را می‌سازد و وصل می‌کند.

**اگر سرویس را دستی ساخته‌اید:**

1. یک دیتابیس **PostgreSQL** بسازید (`New +` → `Postgres`) — منطقه را **همان
   منطقهٔ سرویس وب** انتخاب کنید، وگرنه اتصال داخلی کار نمی‌کند.
2. در سرویس وب → `Environment`، این‌ها را تنظیم کنید:
   - `DATABASE_URL` = مقدار **Internal Database URL** دیتابیس.
   - `JWT_SECRET` = یک مقدار تصادفی ۶۴ کاراکتری.
   - `ADMIN_PASSWORD` = رمز دلخواه مدیر.
3. سلامت اتصال را از این آدرس بررسی کنید: `/api/health`
   (بدون فاش کردن رمز، می‌گوید Postgres وصل شده یا نه).

> ⚠️ **بدون `DATABASE_URL`، داده‌ها ماندگار نیستند.** فایل‌سیستم Render موقتی است؛
> با هر دیپلوی، دیتابیس SQLite و پوشهٔ `uploads/` پاک و از نو ساخته می‌شوند.

## امنیت

- رمزها با **bcrypt** (cost ۱۲) هش می‌شوند؛ هرگز به‌صورت متن ذخیره یا برگردانده نمی‌شوند.
- **JWT** برای احراز هویت؛ نقش و وضعیت فعال بودن در هر درخواست از دیتابیس **دوباره
  خوانده** می‌شود، پس تنزل سطح یا غیرفعال‌سازی بلافاصله اثر می‌کند.
- **محافظت brute-force:** پس از ۳ تلاش ناموفق کپچای ساده، پس از ۵ تلاش بلاک ۱۵ دقیقه‌ای
  روی **هم نام کاربری و هم IP**.
- **پاک‌سازی ورودی:** تمام HTML کاربر با `sanitize-html` (سیاست allowlist) پاک می‌شود؛
  الگوهای حمله (XSS/SQLi/traversal) شناسایی و به‌عنوان رویداد امنیتی برای مدیر ثبت می‌شوند.
- **کوئری پارامتری** در همه‌جا (دفاع اصلی در برابر SQL injection).
- **CSP سخت‌گیرانه:** اسکریپت فقط از خود سایت (`script-src 'self'`)، بدون اسکریپت درون‌خطی.
- لینک ویدیو فقط از دامنه‌های سفیدفهرست‌شده؛ لینک‌های SSRF رد می‌شوند.
- آپلود: محدودیت حجم و نوع، بررسی **امضای واقعی فایل** (نه فقط پسوند)، نام امن.
- محدودیت نرخ درخواست، هدرهای امنیتی helmet، لاگ ممیزی رویدادهای حساس.

## تست‌ها

```bash
npm test                 # تست‌های آفلاین: تاریخ شمسی، پاک‌سازی، لایهٔ دیتابیس

# تست‌های یکپارچه (نیازمند سرور در حال اجرا روی همان پورت):
npm run test:permissions # جداسازی سطوح دسترسی، مالکیت پست، قفل شدن
npm run test:accounts    # ثبت‌نام، پروفایل، درخواست نویسندگی، brute-force
npm run test:api         # ذخیرهٔ محتوا، پاک‌سازی XSS، اعتبارسنجی ویدیو
```

جمعاً بیش از ۱۴۰ مورد تست.

## ساختار پوشه‌ها

```
├── server.js            # نقطهٔ ورود، میدلورها، مسیریابی، مدیریت خطا
├── config.js            # پیکربندی مرکزی و اعتبارسنجی اسرار
├── database.js          # راه‌اندازی SQLite (sql.js) + اجرای مهاجرت‌ها
├── db/index.js          # لایهٔ دسترسی دوگانه Postgres/SQLite
├── migrations.js        # مهاجرت‌های افزایشی اسکیما
├── seed.js              # دادهٔ اولیه: دسته‌بندی‌ها، عمل‌ها، حساب مدیر
├── seed-content.js      # محتوای ۱۸ عمل اول
├── routes/api.js        # تمام مسیرهای API
├── lib/
│   ├── sanitize.js      # پاک‌سازی HTML و شناسایی حمله
│   └── auth-guard.js    # brute-force، کپچا، اعتبارسنجی رمز
├── public/              # فرانت‌اند سایت عمومی
│   ├── index.html · js/app.js · css/style.css
│   ├── js/jalali.js · js/editor.js · vendor/quill/
│   └── 404.html · 500.html
├── admin/index.html     # پنل مدیریت (تک‌صفحه‌ای)
├── scripts/             # اسکریپت‌های تست و ابزار
└── CHECKLIST.md         # ردیاب پیشرفت پروژه (چندجلسه‌ای)
```
